# OpenAPI / REST Contract: JSON Column Surface

**Entity**: `Profile` (table `dbo.profiles`)
**REST base path** (test fixture): `/api/Profile`
**Scope**: How a SQL Server `JSON` column appears in the OpenAPI
schema generated by DAB, and the exact request/response shapes for
every operation defined by the spec's user stories.

This document defines the **client-observable contract**. Internal
serializer behavior may evolve as long as these shapes hold.

---

## Schema component (OpenAPI 3)

DAB emits the JSON column as a plain string property — no `format`,
no `pattern`, no `x-dab-` extension (FR-001, FR-013).

```yaml
components:
  schemas:
    Profile:
      type: object
      properties:
        id:
          type: integer
          format: int32
        metadata:
          type: string
          nullable: true
      # 'id' is omitted from 'required' for create payloads (IDENTITY).
```

## Operations

For brevity, requests and responses below show only the JSON-relevant
fields. Standard DAB response envelopes (e.g., `value[]` for collection
GETs) are unchanged.

### GET /api/Profile/id/1

**Response 200**

```json
{
  "value": [
    {
      "id": 1,
      "metadata": "{\"role\":\"admin\",\"tier\":3}"
    }
  ]
}
```

`metadata` is a JSON **string** whose characters are the JSON text
returned by `SELECT metadata FROM profiles WHERE id = 1`. DAB does not
parse it into a nested object (FR-002, User Story 2).

### GET /api/Profile

**Response 200**

```json
{
  "value": [
    { "id": 1, "metadata": "{\"role\":\"admin\",\"tier\":3}" },
    { "id": 2, "metadata": "{\"tags\":[\"a\",\"b\",\"c\"]}" },
    { "id": 3, "metadata": "{\"nested\":{\"key\":{\"deep\":true}}}" },
    { "id": 4, "metadata": "{\"unicode\":\"éü😀\"}" },
    { "id": 5, "metadata": null }
  ]
}
```

`$select=metadata` and pagination behave identically to any other
nullable string column (User Story 3).

### POST /api/Profile

**Request body**

```json
{
  "metadata": "{\"role\":\"guest\"}"
}
```

**Response 201**

```json
{
  "value": [
    { "id": 6, "metadata": "{\"role\":\"guest\"}" }
  ]
}
```

**Rejected — non-string token on input** (FR-004)

```json
{ "metadata": { "role": "guest" } }
```

→ Response `400 Bad Request`. Rejection is produced by the existing
JSON deserializer for string-typed properties; DAB does not add a
JSON-specific check.

**Rejected — malformed JSON text** (FR-007)

```json
{ "metadata": "{not valid json" }
```

→ Response `400 Bad Request`. SQL Server raises a JSON-validation
error (currently numbers 13608–13614); DAB's existing
`MsSqlDbExceptionParser` maps the error number to 400 once the
numbers are appended to its `BadRequestExceptionCodes` list. The
response body includes the SQL Server error number.

### PUT /api/Profile/id/1

**Request body**

```json
{ "metadata": "{\"role\":\"owner\"}" }
```

**Response 200** — full row with new `metadata` (User Story 5).

### PATCH /api/Profile/id/1

**Request body**

```json
{ "metadata": "{\"role\":\"owner\"}" }
```

**Response 200** — same as PUT for the touched field; other columns
unchanged (User Story 5).

**Clear to NULL** (User Story 8)

```json
{ "metadata": null }
```

→ `metadata` becomes SQL `NULL`; response renders `"metadata": null`.

### DELETE /api/Profile/id/1

**Response 204** (User Story 6). No special handling for JSON.

## Filter & orderby contract (User Story 9, FR-009 / FR-010)

### `$filter` operators on JSON columns

DAB does not maintain a JSON-specific operator allow-list. Every
operator the client supplies is forwarded to SQL Server. The table
below lists what SQL Server itself currently supports and rejects on
the native JSON type; the contract holds as long as SQL Server
behavior holds, and DAB automatically picks up any future operator
support SQL Server adds (FR-009).

| Operator | Forwarded to SQL | SQL Server result (current) | DAB response |
|----------|------------------|-----------------------------|--------------|
| `eq` | Yes | Compares stored JSON text as a string | 200 OK with matching rows |
| `ne` | Yes | Compares stored JSON text as a string | 200 OK with matching rows |
| `eq null` | Yes | Translates to `IS NULL` | 200 OK with matching rows |
| `ne null` | Yes | Translates to `IS NOT NULL` | 200 OK with matching rows |
| `gt`, `lt`, `ge`, `le` | Yes | SQL Server raises a JSON-comparison error | 400 with the SQL error number in the body (FR-007) |
| `contains`, `startswith`, `endswith` | Yes | SQL Server raises a JSON-comparison error | 400 with the SQL error number in the body (FR-007) |
| JSON-path predicates (`JSON_VALUE(...)`, `$.foo`) | Out of scope | Not generated by DAB's OData translator | OData parser may reject upstream of DAB |

**Example — accepted by SQL Server**

```
GET /api/Profile?$filter=metadata eq '{"role":"admin","tier":3}'
GET /api/Profile?$filter=metadata eq null
```

**Example — rejected by SQL Server**

```
GET /api/Profile?$filter=contains(metadata,'admin')
```

→ `400 Bad Request`. Body contains the SQL Server error number that
was raised. DAB does not pre-emptively reject this request — it is
forwarded to SQL Server and the rejection is surfaced verbatim through
the existing error pipeline.

### `$orderby` on JSON columns

Forwarded to SQL Server. Ordering succeeds if SQL Server supports it
on the column and returns its standard string-order result (FR-010).

```
GET /api/Profile?$orderby=metadata
GET /api/Profile?$orderby=metadata desc
```

## Error envelope summary

| Cause | HTTP status | DAB sub-status |
|-------|-------------|----------------|
| `metadata` sent as a non-string token (FR-004) | 400 | BadRequest (from JSON deserializer for string-typed property) |
| Malformed JSON text in `metadata` (FR-007) | 400 | BadRequest (SQL Server JSON-validation error number appended to `MsSqlDbExceptionParser.BadRequestExceptionCodes`) |
| `$filter` operator that SQL Server does not support on the JSON type (FR-009) | 400 | BadRequest (same parser path as above; response body contains the SQL error number) |
| Target SQL Server build lacks JSON type (FR-016) | 500 at startup, or upstream SQL error at request time | (unmapped — native SQL Server error surfaces) |
| Permission denied | 403 | unchanged |
| Row not found | 404 | unchanged |
| Delete on row with JSON | 204 | unchanged |
